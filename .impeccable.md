## Design Context

### Users
CarPulse is for independent car resellers and small dealerships in the Baltics and Finland. Many users are middle-aged or older, check listings repeatedly throughout the day, and care more about speed, clarity, and trust than novelty. Their main job is to define a buying radar, notice a suitable vehicle quickly, and open the original listing before competitors do.

### Brand Personality
Strict, familiar, dependable. CarPulse should feel like established European automotive software: commercially serious, easy to scan, and calm under pressure.

### Aesthetic Direction
A light, professional interface inspired by the clarity and familiarity of mobile.de without copying its visual identity. Use opaque surfaces, strong borders, restrained red as the single accent, small 8–10px radii, high-contrast typography, and real vehicle photography. Avoid neon, glassmorphism, translucent panels, large decorative gradients, excessive shadows, oversized radii, and generic futuristic SaaS styling.

### Design Principles
1. Put fresh vehicles, price, and detection time before dashboard decoration.
2. Keep forms explicit: visible labels, familiar controls, readable defaults, and manual numeric entry.
3. Design for older users with 16px+ body text, 44px+ targets, strong contrast, and no hover-only actions.
4. Use one restrained accent and let spacing, type weight, and borders create hierarchy.
5. Make system trust visible through source health, last scan time, and clear delivery state.

## Feature Brief

### Feature Summary
The core product is a personal web dashboard where each user creates up to ten vehicle radars, reviews newly matched listings, saves promising cars, and connects a Telegram bot for immediate delivery. Marketplace adapters and notification delivery remain separate backend capabilities so either can evolve without changing the dashboard.

### Primary User Action
Create or refine a radar, then act on a newly detected vehicle from the unified feed.

### Layout Strategy
Use a persistent left navigation on desktop and a compact top navigation on mobile. The main viewport leads with the fresh-listing feed, a concise radar status row, and an obvious “Create radar” action. Vehicle results use horizontal rows on desktop and stacked cards on mobile; filters use a dedicated panel with progressive disclosure for advanced fields.

### Key States
- First run with no radars: explain the value and offer one clear creation action.
- Active monitoring: show source, last scan, active filters, and new-match count.
- New matches: prioritize price, age of listing, vehicle facts, source, and original link.
- Source degraded: state what is delayed, when it last worked, and that other sources continue.
- Telegram disconnected/connected: show a short connection flow and delivery confirmation.
- Loading, validation error, save success, and empty filtered feed states.

### Interaction Model
Users navigate between Overview, Radars, Vehicles, Favorites, and Settings. Radar creation uses standard fields with suggested values while preserving manual entry for price, year, and mileage. Low-risk actions update immediately; destructive actions expose undo. Telegram connection uses a one-time code generated in Settings.

### Content Requirements
Use short, literal Russian labels. Keep marketplace names, prices, mileage, source health, and detection age easy to compare. Error copy must explain what happened and what the user can do next.
